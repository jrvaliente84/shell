# Shell - Frontend Development Guidelines

# Optimización de Diálogo y Tokens
- Antes de generar bloques grandes de código, utiliza la herramienta `askQuestions` para confirmar requisitos ambiguos.
- Si el contexto es insuficiente para una tarea en Angular, detente y solicita los archivos específicos necesarios (ej. el HTML del componente o el archivo de estilos).
- Prioriza respuestas cortas y modulares. No repitas código que ya existe en el archivo actual.
- Usa interacción "Human-In-The-Loop" para validar lógica de negocio compleja antes de implementarla.


## Objetivo
Guía de buenas prácticas para desarrollo frontend en el proyecto shell con enfoque en arquitectura limpia, principios SOLID y Clean Code.

## Principios Fundamentales

### 1. Clean Architecture
- **Separación de capas**: Mantener componentes, servicios, modelos y utilidades en estructura clara
- **Independencia de frameworks**: El código de negocio debe ser independiente de Angular
- **Testabilidad**: Cada componente debe ser fácilmente testeable en aislamiento
- **Bajo acoplamiento, alto cohesión**: Reducir dependencias entre módulos

Estructura recomendada:
```
src/app/
├── core/              # Servicios singleton, guards, interceptors
├── shared/            # Componentes reutilizables, pipes, directives
├── features/          # Módulos de features (lazy-loaded)
│   └── [feature]/
│       ├── models/
│       ├── services/
│       ├── components/
│       └── pages/
└── config/            # Configuración global
```

### 2. Principios SOLID
- **S (Single Responsibility)**: Cada componente/servicio una responsabilidad
- **O (Open/Closed)**: Abierto para extensión, cerrado para modificación
- **L (Liskov Substitution)**: Las subclases deben ser sustituibles
- **I (Interface Segregation)**: Interfaces específicas, no genéricas
- **D (Dependency Inversion)**: Depender de abstracciones, no de implementaciones

### 3. Clean Code
- **Nombres claros y descriptivos**: Variables, funciones, clases con nombres que revelen intención
- **Funciones pequeñas**: Una función = una responsabilidad
- **DRY (Don't Repeat Yourself)**: Evitar código duplicado
- **Manejo de errores explícito**: Try/catch adecuados, validaciones claras
- **Comentarios significativos**: Solo si el código no es autoexplicativo
- **Formato consistente**: 2 espacios indentación, orden de propiedades, etc.

### 4. TypeScript Best Practices
- **Strong typing**: Usar tipos explícitos, evitar `any`
- **Strict mode**: Habilitar en tsconfig.json
- **Null safety**: Validar nulidad de objetos
- **Enums vs strings**: Usar enums para constantes enumeradas

### 5. Angular Best Practices
- **Componentes standalone**: Usar componentes standalone para nuevas features
- **Reactive forms**: Usar Reactive Forms (FormGroup, FormControl) en lugar de Template-driven
- **OnPush change detection**: Usar estrategia OnPush para optimización
- **Memory leaks prevention**: Usar takeUntilDestroyed(), unsubscribe en ngOnDestroy
- **Smart/Dumb components**: Separar lógica (smart) de presentación (dumb)

### 6. Accesibilidad (a11y)
- **ARIA labels**: Añadir etiquetas ARIA en elementos interactivos
- **Semantic HTML**: Usar elementos semánticos (`<nav>`, `<article>`, etc.)
- **Keyboard navigation**: Asegurar navegación completa por teclado
- **Color contrast**: Mantener suficiente contraste (WCAG AA minimum)

### 7. Performance
- **Lazy loading**: Cargar features bajo demanda
- **OnPush strategy**: Change detection manual
- **TrackBy en *ngFor**: Implementar trackBy para listas
- **Code splitting**: Separar código en chunks manejables

## Verificación Obligatoria

### Antes de hacer commit:
```bash
# 1. Compilación
npm run build

# 2. Linting
ng lint

# 3. Formato
npm run format

# 4. Tests (cuando estén disponibles)
npm test
```

### Criterios de aceptación:
- ✅ Código compila sin errores (`npm run build` exitoso)
- ✅ No hay warnings de TypeScript (strict mode activo)
- ✅ Linting pasa sin problemas
- ✅ Código formateado correctamente
- ✅ Tests pasan (cuando estén implementados)
- ✅ No hay console.log() en producción
- ✅ No hay warnings de accesibilidad
- ✅ Bundle size dentro de presupuesto

## Checklist de Revisión de Código

- [ ] ¿El componente tiene una única responsabilidad clara?
- [ ] ¿Los nombres son descriptivos y revelan intención?
- [ ] ¿Hay código duplicado que pueda refactorizarse?
- [ ] ¿Se pueden mejorar los tipos TypeScript?
- [ ] ¿Se manejan los errores apropiadamente?
- [ ] ¿Se previenen memory leaks (observables, listeners)?
- [ ] ¿Es testeable el código?
- [ ] ¿La accesibilidad está considerada?
- [ ] ¿El componente es reutilizable?
- [ ] ¿Compila correctamente sin warnings?

## Design System
- **Colores**: Dark Brutalism con acento oro #d4af37
- **Tipografía**: Space Mono (display), Geist (body)
- **Animaciones**: cubic-bezier(0.4, 0, 0.2, 1), duración 300-500ms
- **Consistencia visual**: Usar variables CSS definidas en :host
